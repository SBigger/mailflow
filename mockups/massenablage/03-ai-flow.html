🤖 Mockup 03 — KI-Ablauf & Lern-Logik

Dies ist der konzeptuelle Ablauf, wenn eine Datei in den Massenablage-Dialog gezogen wird. Nichts davon ist schon gebaut — nur der Plan. Lies ihn durch und sag, wo du etwas anders willst.
1

🗂️ Datei landet im Dialog

Nutzer zieht 1–20 Dateien in die Drop-Zone oder klickt „Auswählen". Jede Datei bekommt sofort eine Zeile in der Liste mit Status ⏳ analysiert…
lokal Preview wird sofort via URL.createObjectURL(file) erzeugt — keine Internet-Verbindung nötig
lokal Datei bleibt im Browser — noch kein Upload
▼
2

Je nach Dateityp andere Methode — alles lokal im Browser:
DateitypMethodeDauer typisch

PDF (Scan / Bild)pdfjs → Canvas → Tesseract.js OCR3–15s pro Seite

Ergebnis: roher Text-String (max. 100'000 Zeichen), gespeichert als content_text (gleiches Feld wie bestehender Upload)

3

🔍 Lokales Matching zuerst (günstig & schnell)

Bevor wir Claude anrufen (kostet Geld), versuchen wir es mit einfachen Regeln:
  • Kunde: UID-Regex CHE-\d{3}\.\d{3}\.\d{3} im Text → direkter Treffer in customers.uid
  • Kunde Fallback: Firmen-Name (aus customers.company_name) als Substring suchen — Groß-/Kleinschreibung egal, Umlaute normalisiert
  • Tag: Keyword-Tabelle aus dok_tag_learnings abfragen — z.B. „MWST-Abrechnung" → Tag MWST › Abrechnung
  • Jahr: Regex \b(20\d{2})\b (ist schon im Projekt, Z. 24 in DokUploadDialog.jsx)
  • Kategorie: aus Tag-Zuordnung ableitbar (jeder Tag hat eine Kategorie)
Treffer Feld wird gelb vorbefüllt, Badge „✨ KI · 95%" → weiter zu Schritt 5
Unsicher Kunde gefunden, Tag nicht → weiter zu Schritt 4 nur für Tag
Nichts Weder UID noch Firma im Text → weiter zu Schritt 4 (Claude entscheidet)
▼
4

🧠 Claude-Fallback (nur wenn lokal nicht reicht)

Supabase Edge Function suggest-document-fields (neu) ruft Claude API:
  • Input: Datei-Text (erste 10'000 Zeichen), Liste aller Kunden (ID + Name + UID), Liste aller Tags (ID + Name + Parent + Kategorie)
  • Output: {customer_id, tag_ids[], category, year, name, confidence}
  • Caching: Antwort per SHA-256-Hash des Datei-Inhalts cachen → gleiche Datei wird nie 2× bezahlt
  • Kosten: ~0.003 CHF pro Datei (Claude Haiku), ~0.015 CHF (Sonnet)
Confidence ≥ 80% Felder werden vorbefüllt (gelb)
50–80% Felder vorbefüllt mit Warn-Badge „niedrige Sicherheit, bitte prüfen"
< 50% / kein Kunde Datei wird geparkt → rotes Badge, bleibt im Dialog liegen
▼
5

👤 Nutzer prüft, korrigiert, klickt „Hochladen"

Alle KI-Felder sind editierbar (Dropdowns aus Stammdaten). Nutzer kann
  • einzelne Felder überschreiben
  • mehrere Dateien auf einmal auswählen und Batch-Änderung vornehmen (z.B. „alle auf Kunde X zuweisen")
  • geparkte Dateien jetzt manuell vervollständigen
  • einzelne Dateien entfernen
Klick auf „X Dateien hochladen" → Batch-Upload in den Bucket dokumente, pro Datei ein Insert in dokumente-Tabelle. Exakt gleich wie der bestehende Einzel-Upload, nur in Schleife.
▼
6

📚 Lernen: was der Nutzer korrigiert, wird gespeichert

Wenn der Nutzer einen KI-Vorschlag ändert, lernen wir daraus:

🧠 Learning-Tabelle (neu, additiv)

Eine neue Tabelle dok_tag_learnings merkt sich Keyword→Tag-Paare aus Nutzer-Korrekturen. Kein Overhead für die KI, reines Nachschlagen beim nächsten Match.

create table dok_tag_learnings (
  id           uuid primary key default gen_random_uuid(),
  keyword      text not null,                    -- z.B. "lohnjournal", "mwst-abrechnung"
  tag_id       uuid references dok_tags(id) on delete cascade,
  customer_id  uuid references customers(id) on delete set null,  -- optional: kundespezifisch
  hits         int default 1,                    -- wie oft bestätigt
  source       text default 'user_correction',  -- woher kam das Learning
  created_at   timestamptz default now()
);

create index dok_tag_learnings_keyword_idx on dok_tag_learnings(lower(keyword));

Beispiel: Nutzer hat „Rechnung_Bühler_2026.pdf" korrigiert von „Rechnungswesen › Allgemein" auf „Rechnungswesen › Ausgangsrechnung" → neuer Eintrag ("rechnung", tag_id_ausgangsrechnung, bühler_id, hits=1). Beim nächsten Upload für Bühler mit „Rechnung" im Namen trifft es sofort richtig.

🅿️ Sonderfall: kein Kunde gefunden

Nicht blockieren — parken.

Die Datei bleibt im Dialog mit Badge ⚠ kein Kunde. Nutzer kann sie später manuell zuweisen ODER mit „Abbrechen" den Dialog schließen — die anderen Dateien sind bereits hochgeladen, die geparkte bleibt erhalten (im nächsten Öffnen des Dialogs wird sie wieder angezeigt via LocalStorage-Cache oder Session-State). Kein Datenverlust, kein „Alles oder nichts".

🔒 Was garantiert nicht passiert

  • Check-out/-in unberührt: Dieser Dialog ruft handleCheckout / handleCheckin nie auf und schreibt nie in checked_out_by, checked_out_at
  • Bestehende Upload-Dialoge bleiben: Die alten Einzel-Upload-Dialoge (DokUploadDialog.jsx, inline UploadDialog in Dokumente.jsx) werden nicht verändert
  • Live-Seite smartis.me: Wir arbeiten auf Branch feat/massenablage — solange kein git push zu master passiert, geht auf Vercel nichts live
  • Datenbank: Nur additive Migration (dok_tag_learnings-Tabelle neu), keine Änderung an dokumente-Schema, keine Änderung an customers, dok_tags
  • MS365: Dieser Dialog spricht nicht mit MS365. Punkt.
